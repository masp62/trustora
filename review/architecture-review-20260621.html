Problem

Post validation logic is defined in 4 places — 3 server-side copies in post-actions.ts and one client copy in the create form — with diverging error messages already.

Solution

One validatePostInput(fields, mode) module — mode: 'draft' | 'publish' controls which rules apply. All 4 call sites import it.

Wins

  • ✦ locality: new rule added once
  • ✦ interface shrinks — callers get errors, not conditionals
  • ✦ tests cross one seam, not four
  • ✦ delete 3 shallow blocks → complexity concentrates

C-2 · Deepen PostCardData assembly into one module

src/lib/explore-feed.ts

location-feed.ts · PostCardData assembly

→ 60 DB round-trips per page (3 per post × 20 posts), repeated 5 times

After — one deep assembler
interface
assembleFeedCards(posts[], viewerId)
implementation
· batch author lookup (1 query)
· batch cover image lookup (1 query)
· batch like counts (1 query)
· batch viewer liked-status (1 query)
· assemble PostCardData[]

→ 4 batched queries per page regardless of feed

Problem

Five feeds each copy the same per-post author/image/likeCount Promise.all loop — 60 individual DB round-trips per page load. The PostCardData shape is implicitly defined by whoever assembles it; silently different between feeds.

Solution

One assembleFeedCards(posts, viewerId) module that batches all lookups into 4 queries and returns a typed PostCardData[]. Every feed calls it; none assembles inline.

Wins

  • ✦ leverage: 5 feeds share one deep module
  • ✦ locality: N+1 fix is in one place
  • ✦ PostCardData shape is the interface, not implicit
  • ✦ tests cross one seam with a DB adapter

Problem

The in-memory adapter is a 500-line re-implementation of Prisma's query engine. It silently diverges — no support for OR, nested relations, or complex where. Code that passes the in-memory emulator can fail against Postgres in production.

Solution

Delete createInMemoryDb. Run tests against a real Postgres instance (Docker, pg test container, or Neon branch). Use transaction rollback or prisma migrate reset per test suite for isolation.

Wins

  • ✦ delete 500 lines of shallow pass-through
  • ✦ locality: schema changes need one migration, not two
  • ✦ tests cross the real persistence seam
  • ✦ OR, joins, full-text — all testable

C-4 · Fix explore-feed's in-memory sort-then-slice pagination

src/lib/explore-feed.ts
Before — full table, JS pagination
sequenceDiagram
  participant C as Caller
  participant F as explore-feed.ts
  participant DB as Postgres
  C->>F: getExploreFeed(cursor, page=2)
  F->>DB: findMany(all published posts, no LIMIT)
  DB-->>F: 5000 rows
  F->>DB: findMany(all likes for 5000 posts)
  DB-->>F: 50000 rows
  F->>DB: findMany(all comments for 5000 posts)
  DB-->>F: 20000 rows
  Note over F: sort by engagementScore in JS
  Note over F: slice to 20
  F-->>C: PostCardData[20]
After — DB-side sort + LIMIT
sequenceDiagram
  participant C as Caller
  participant F as explore-feed.ts
  participant DB as Postgres
  C->>F: getExploreFeed(cursor, page=2)
  F->>DB: findMany(published, ORDER BY score, SKIP cursor, LIMIT 21)
  DB-->>F: 21 rows
  F->>DB: assembleFeedCards(21 posts)
  DB-->>F: 4 batched queries
  F-->>C: PostCardData[20] + nextCursor

Problem

The module exposes a cursor-pagination interface but the implementation loads every matching post into Node.js memory, sorts by engagement score in JS, then slices. The cursor encodes a transient JS sort position and becomes invalid as new likes arrive mid-scroll.

Solution

Persist engagementScore to the DB (updated by a trigger or on-write). Sort and paginate via DB orderBy + skip/take. The caller interface doesn't change — only the implementation deepens.

Wins

  • ✦ depth: large complexity hidden behind existing interface
  • ✦ locality: pagination semantics in one place
  • ✦ O(page size) rows instead of O(all posts)
  • ✦ cursor becomes stable across requests

Problem

isBanned is checked in 2 of 7 write actions. Banned users can still like content, follow users, and edit posts. Any new write action will silently miss the check — the shallow per-action pattern guarantees it.

Solution

One requireActiveUser(session) module: checks auth + isBanned + optional role. All write actions call it first. Alternatively — store isBanned in the JWT token to avoid the extra DB lookup.

Wins

  • ✦ locality: enforcement in one module
  • ✦ leverage: N actions, one check to test
  • ✦ new actions can't forget the check
  • ✦ JWT option: eliminates DB round-trip per request

C-8 · Push search filtering into the DB — remove full table scan

src/lib/search-feed.ts
Before — O(n posts) per query in Node.js
interface: searchPosts(viewerId, query) → PostCardData[]
implementation:
load ALL published posts
→ filter in JS
→ sort by relevance in JS
→ N+1 assembly loop
(same hidden cost as C-2)

O(all posts) rows into Node.js memory per search

After — DB-side pre-filter, JS relevance re-rank
interface unchanged: searchPosts(viewerId, query)
implementation
· DB: title/body ILIKE %query% (indexed)
· JS: re-rank small candidate set
· assembleFeedCards() (batched)

O(matches) rows — 90%+ reduction at scale

Problem

Every search loads every published post into Node.js memory. The computeRelevance function is pure and testable, but the bug it hides is in the call site: there's no DB-side filtering at all.

Solution

Add a DB-side pre-filter (ILIKE or Postgres full-text search) to shrink the candidate set before JS re-ranking. Interface stays the same; implementation deepens behind the seam.

Wins

  • ✦ depth: same interface, far less surface overhead
  • ✦ leverages DB index — O(matches) not O(posts)
  • ✦ composable with C-2: feed card assembly batched
  • ✦ computeRelevance stays testable and pure

Top recommendation

Start with C-2 + C-1

C-2 (assembleFeedCards) is the highest-leverage move: it gives you the deepened persistence seam that C-4 and C-8 both depend on, and it stops the N+1 query pattern from being re-introduced whenever a new feed is added. It's also entirely mechanical — no domain decisions, just extracting what already exists in 5 identical places. Deletion test passes clearly.

C-1 (post validation module) pairs naturally with C-2 as a same-session refactor. It eliminates the highest risk of correctness divergence in the codebase (validation rules defined 4 times, already drifting) and creates a clean seam for publishing-mode rules. Together the two moves delete roughly 200 lines of copy-paste and concentrate the most important business rules in testable modules.

→ C-2 · assembleFeedCards highest leverage, immediate gain
→ C-1 · validatePostInput correctness risk, clean seam
C-3 (drop in-memory emulator) is a prerequisite for C-4 and C-8 at any real scale — defer until test infrastructure is ready for a real Postgres test database.
